# 聆听 v0.6 页面重构设计

## 背景

v0.6 是聆听从 MVP 闭环进入成熟产品方向的第一阶段。当前后端、SQLite 数据模型、扫描链路、播放流和 HTTP API 已经可以支撑本地单用户音乐管理，页面重构不以扩展后端能力为目标。

本阶段重点是重组前端信息架构，让界面从“功能拼接”变成可以长期使用和继续扩展的桌面音乐产品。

## 主界面方向

v0.6 固化“高级无右栏版”为主界面方向：

- 左侧导航承担全局位置感和页面切换。
- 中间主工作区根据当前页面展示最相关任务。
- 底部播放器承担全局播放控制。
- 播放队列只通过播放器入口打开，不常驻为右侧信息栏。
- 不保留右侧重复信息列，避免当前播放、扫描状态、队列预览相互抢占空间。

## 页面职责

### 歌曲页

歌曲页是默认主工作区。它负责展示全局歌曲库、当前播放摘要、搜索结果、筛选、排序、批量操作和播放入口。

歌曲页不管理媒体库扫描细节，也不常驻播放队列。它应该保持高频浏览和播放操作的效率。

### 媒体库页

媒体库页专门负责本地目录管理：

- 添加媒体库目录。
- 展示每个媒体库的歌曲数、扫描状态、进度和消息。
- 触发重扫。
- 删除媒体库索引。

扫描状态不再散落到其他页面，只通过全局摘要和媒体库页呈现。

### 歌单页

歌单页负责普通歌单的创建、选择、重命名、删除和歌曲管理。普通歌单不混入“我喜欢”和“最近播放”，系统歌单继续作为独立导航项。

### 系统歌单

“我喜欢”和“最近播放”复用成熟列表页结构，保留播放、批量操作、添加到歌单等能力。“最近播放”保留清空入口。

### 播放器与队列

底部播放器始终展示当前歌曲、播放状态、进度、播放/暂停、上一首/下一首、播放模式、音量和队列入口。

队列作为抽屉从播放器打开，展示当前播放和接下来歌曲。队列抽屉不替代任何页面职责。

## 实现边界

- 不改 Go 后端 API。
- 不改 SQLite schema。
- 不新增封面、专辑页、艺术家页或桌面端能力。
- 不引入完整 E2E 栈。
- 使用轻量前端测试锁定播放、队列、排序和本地存储兼容逻辑。

## 验收标准

- 前端品牌显示为“聆听”。
- 首屏默认进入歌曲页。
- 媒体库页只承载目录和扫描管理。
- 队列只通过播放器入口打开。
- 原有核心能力完整保留。
- `npm run build` 通过。
- `go test ./...` 通过。
